# 🚀 Kong API Deployment using Terraform + OpenAPI + decK

## 📌 Overview

This project automates the deployment of APIs to **Kong Konnect** using a fully automated pipeline:

> OpenAPI → decK → Kong YAML → Python Transformer → Terraform → Kong Konnect

It ensures infrastructure is **repeatable, declarative, and idempotent-ready (with uniqueness handling).**

---

## 🧱 Architecture

```
OpenAPI Spec (openapi.yaml)
        ↓
decK (openapi2kong)
        ↓
kong.yaml
        ↓
convert.py
        ↓
Terraform JSON (with plugins enabled)
        ↓
Terraform Apply
        ↓
Kong Konnect
   ├── Service
   ├── Routes
   ├── key-auth Plugin
   └── rate-limiting Plugin
```

## 🚀 How It Works

### 1. OpenAPI → Kong (decK)

```bash
deck file openapi2kong --spec openapi.yaml --output-file kong.yaml
```

👉 Output format used by this project:

```yaml
services:
  - name: httpbin-api-via-kong
    host: httpbin.org
    protocol: https
    port: 443
    path: "/"
    routes:
      - name: httpbin-api-via-kong_getanything
        paths:
          - ~/anything$
        methods:
          - GET
```

---

### 2. Kong → Terraform JSON

```bash
python convert.py
```

## 🔁 High-Level Flow of convert.py

`convert.py` is responsible for transforming **Kong decK-generated YAML (kong.yaml)** into a **Terraform-ready JSON configuration** that can be directly deployed to Kong Konnect.

---

### 📥 1. Input Stage (Kong YAML)

The script reads `kong.yaml` generated by decK, which contains:

- Services (host, protocol, port)
- Routes (paths, methods)
- Service-to-route relationships

This file acts as the **source of truth after OpenAPI conversion**.

---

### 🔍 2. Format Detection & Validation

The script first validates the structure:

- Ensures `services` exist in YAML
- Rejects unsupported or malformed configurations
- Validates required fields like:
  - service name
  - protocol
  - host
  - port

---

### 🔄 3. Service Transformation

Each service is converted into a Terraform resource:

- Maps Kong service → `konnect_gateway_service`
- Normalizes service names (safe naming convention)
- Injects default routing path:
  - `path: "/"`

This ensures all services have a valid base path even if missing in input.

---

### 🔐 4. Plugin Injection (Service Level Security Layer)

For every service, two default plugins are automatically attached:

#### 🔑 Key Authentication Plugin

- Enforces API key validation
- Protects service access
- Requires `apikey` header from consumers

#### 🚦 Rate Limiting Plugin

- Limits API usage per consumer
- Default policy:
  - 5 requests per minute
- Prevents abuse and overload

---

### 🛣️ 5. Route Transformation

For each service:

- Routes are extracted and converted into Terraform resources
- Maps:
  - paths
  - HTTP methods
  - service association

Each route is linked to its parent service using Terraform references.

---

### 🧠 6. Naming & Uniqueness Strategy

To ensure safe Terraform execution:

- Service names are normalized (lowercase, safe characters only)
- Unique suffixes are added to avoid collisions
- Route names are also uniquely generated

This guarantees:

- Idempotent deployments
- No duplicate resource conflicts

---

### ⚙️ 7. Terraform JSON Generation

After processing all resources, the script builds:

- `generated.tf.json`

This file contains:

- Kong services
- Kong routes
- key-auth plugin configurations
- rate-limiting plugin configurations

Fully compatible with Terraform Konnect provider.

---

### 📤 8. Output Stage

Final output is a **ready-to-deploy Terraform JSON file**, which is applied using:

````bash
terraform apply

### 3. Deploy using Terraform

```bash
terraform init
terraform validate
terraform apply
````
